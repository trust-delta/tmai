# tmaiの強み

tmaiが得意なこと、他のツールにはない特徴。

## 1. 1パネル完結の操作

tmaiは**アタッチ不要**で承認操作ができます。

```
┌─────────────────────────────────────────────────────────────┐
│ 一般的なツール                                               │
│                                                             │
│  監視 → アタッチ → y入力 → デタッチ → 監視                 │
│         (3ステップ)                                         │
│                                                             │
│  問題: 操作中は他のエージェントが見えない                   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ tmai                                                        │
│                                                             │
│  監視しながら y キー                                        │
│  (1ステップ)                                                │
│                                                             │
│  利点: 全エージェントを見ながら即座に対応                   │
└─────────────────────────────────────────────────────────────┘
```

**効果**:
- 3エージェント連続承認: 他ツール9操作 → tmai 6操作
- コンテキストスイッチなし
- 緊急対応時も全体が見える

## 2. AskUserQuestion完全対応

Claude Codeの `AskUserQuestion` ツールに対応。

```
┌─────────────────────────────────────────────────────────────┐
│ Claude Codeの質問                                           │
│                                                             │
│  Which approach do you prefer?                              │
│                                                             │
│  1. Use async/await                                        │
│  2. Use callbacks                                          │
│  3. Use promises                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘

tmaiでの操作: 「1」キーを押すだけ
```

- 番号キー（1-9）で直接選択
- 全角数字（１-９）にも対応
- 複数選択（multi_select）もSpaceキーでトグル

**他のツール**: 多くはy/n承認のみ、選択肢への対応なし

## 3. 導入障壁の低さ

tmaiは**既存のワークフローを変えずに**導入できます。

```
┌─────────────────────────────────────────────────────────────┐
│ 他のツールの導入                                             │
│                                                             │
│  今までの方法:                                               │
│    tmux → claude                                            │
│                                                             │
│  ツール導入後:                                               │
│    ツール専用コマンドでセッション作成                        │
│    → ワークフローの変更が必要                               │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ tmaiの導入                                                   │
│                                                             │
│  今までの方法:                                               │
│    tmux → claude                                            │
│                                                             │
│  tmai導入後:                                                 │
│    tmux → claude (そのまま)                                 │
│    + 別ペインで tmai                                        │
│                                                             │
│  → ワークフロー変更なし                                     │
└─────────────────────────────────────────────────────────────┘
```

**効果**:
- 「とりあえず監視だけ」から始められる
- チームへの段階的な導入が可能
- 既存の環境・スクリプトをそのまま活用

## 4. 柔軟なワークツリー対応

tmaiは「監視に徹する」ため、ワークフローを強制しません。

- ワークツリーを使っても使わなくてもOK
- 動かしながら追加・削除が自由
- 自分のやり方で進められる

詳細: [ワークツリーで並列開発](../workflows/worktree-parallel.md)

## 5. PTYラッピングによる高精度検出

`tmai wrap claude` で起動すると、PTYプロキシ経由で入出力を直接監視。

```
┌─────────────────────────────────────────────────────────────┐
│ 従来方式 (capture-pane)                                      │
│                                                             │
│  tmux capture-pane → 画面の文字列をパース → 状態推定        │
│                                                             │
│  問題: タイミングによっては状態を見逃す可能性               │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ PTYラッピング                                                │
│                                                             │
│  入出力を直接監視 → リアルタイムで状態検出                  │
│                                                             │
│  利点: 状態遷移を見逃さない、より正確な検出                 │
└─────────────────────────────────────────────────────────────┘
```

詳細: [PTYラッピング](../features/pty-wrapping.md)

## 6. 外部送信検知（セキュリティ）

AIエージェントによる機密情報の外部送信を検知・ログ記録。

検知対象:
- HTTPリクエスト（curl, wget等）
- ファイル転送（scp, rsync等）
- クラウドCLI（aws, gcloud等）
- APIキー、認証情報のパターン

```
INFO  External transmission detected command="curl" pid=12345
WARN  Sensitive data in transmission command="curl" sensitive_type="API Key"
```

詳細: [外部送信検知](../features/exfil-detection.md)

## 7. Web Remote Control

スマホからQRコードをスキャンして操作可能。

- エージェント一覧表示
- y/n承認
- AskUserQuestion選択
- テキスト入力

外出先や別の場所からでも承認操作ができます。

詳細: [Web Remote Control](../features/web-remote.md)

## まとめ

| 強み | 説明 |
|------|------|
| 1パネル完結 | アタッチ不要で即座に操作 |
| AskUserQuestion | 番号キーで選択肢を直接選択 |
| 導入障壁の低さ | 既存ワークフローを変えずに導入 |
| 柔軟性 | ワークツリーも自由に使える |
| 高精度検出 | PTYラッピングでリアルタイム検出 |
| セキュリティ | 外部送信検知 |
| リモート操作 | スマホから承認可能 |
