# Aim — 目的地と、着いたかどうか

tmai のコンソールは、ツールが代われない二つのことをあなたに求める: ある仕事が*どこへ向かうか*を言うこと、そして*そこに着いた*と言うこと。**Aim** は、その両方を tmai が現在記録する方法だ。順序が大事だ: Aim が先にあって tmai がその周りに建てられたのではない。**先にあったのは原則で、Aim はその原則が今のところ手に取っている手段** ― うまくいくから使われていて、より良い手段が現れた日には差し替わる。

これは [README](../README.ja.md) の「交差点を渡る」の絵の続きだ: 目的地を決めるのはあなた、走るのはエージェント、そして信号が決してあなたの代わりに言えない一言が*「あなたは着いた」*。Aim は、その目的地が書き留められ、その到着が宣言される場所だ。

## Aim が仕える原則

tmai は一つの賭けの上に建っている: 希少なのはあなたの判断であって、エージェントの計算量ではない([charter](decisions/2026-05-18-public-tmai-charter.md))。その判断のうち二つの act は分解不能にあなたのもので、どれだけ強いモデルでも取り除けない:

- **目的地を名指す** ― あなたがコミットする outcome、間違っていたら引き受けることになるもの
- **到着を宣言する** ― 仕事が本当にそこへ着いたと判断すること。本物の検証に裏打ちされ、ちらっと見や green check ではなく

この賭けを尊重するツールは、その両方を置く場所を要る ― 目的地を*チェックリスト*でなく*人間のコミットメント*のまま保ち、到着を*自動 pass* でなく*人間の act* のまま保つ場所。そこは done 宣言を **locatable**(一つの場所・一人の持ち主)・**obligatory**(それ無しには ship できない)・**traceable**(何を検証したかの記録が残る)にする ― だが決して*あなたの代わりに判断は下さない*。Aim がその場所だ。

## Aim は前提ではなく手段

Aim が公理でなく導かれたものだという最も明快な証拠は、tmai が**以前の形を置き換えて**この形に至ったことだ。最初の装置は一対の記録だった: **decision**(あなたが引き受ける outcome)を、**approach**(今それをどう追っているか・可変)から意図的に離して保つ。それら退役した記録は今も [`docs/archive/`](archive/) で読める ― 現行の方向ではなく監査証跡として残してある。Aim はその後継だ: 「目的地」と「どう追っているか」を一つの進化する記録に畳み、二スロット形より入れ子の purpose の木にうまくスケールした。

その載せ替えを通じて*変わらなかった*のが、上記の原則だ。目的地は人間のコミットメントのまま、到着は人間の act のまま。それこそが Aim を*手段*と呼ぶことの全てだ: 思想が不動の部分で、装置はその下で動いてよい。明日もっと良く「目的地 + 到着」を表せるものが出れば、Aim もまた置き換わり、load-bearing なものは何一つ動かない。

## aim とは何か

一つの aim は三つの部分を持つ小さな記録で、その分割は装飾ではない ― それが*権威の順序そのもの*だ:

- **目的地** ― この仕事が何の*ため*かを述べる一文、あなたが書く。「X が作られたら done」ではなく、何が*あるべき*かとして書かれる。(目的地が十分条件たる機構として書かれた瞬間、それは静かに spec になっていて、人間のコミットメントは抜け落ちている。)
- **手段** ― Producer が保守する body: 今どう追っているか、何をやったか、何が残っているか。これはエージェントが draft し改訂するもの。その中のマークは Producer が*実装した*ことを記録する ― checkable で fallible な主張 ― のであって、*purpose* が満たされたことは決して記録しない。
- **state** ― open か done か。あなただけが、しかも到着を検証した後にだけ、これを倒す。

だから権威は **目的地(あなた)→ 手段(Producer)→ コード** と流れる。手段は自由に書き換えてよい; コードは現状にすぎず drift しうる。どちらも aim の purpose が満たされたと認証はできない。あなたが state を倒すこと、それだけがそれを行う。

## Aim が越えない一線

Aim は done 宣言を locatable・obligatory・traceable にする。だがそれを *genuine* にはしない ― そして意図的に一歩手前で止まる。purpose が本当に実現されたかは、システムが検査できる条件には形式化しない。なぜならそうすると、そのものが*何であるか*が変わってしまうからだ: 引き受けられた判断をチェックリストに変えたものは、もう判断ではなく、チェックリストへの最適化はそれが代理していたものを静かに消す。これは README の信号と同じ seam だ ― 構造はあなたが見るべき交差点を示せる; だが*「あなたは着いた」*とは決して言えない。その判断は、方針でなく構造上、あなたのものに保たれる。

## エンジンがそこから導くもの ― そしてなぜ how が private か

aim が構造化されているから、エンジンはその上で計算でき、それがこの構造を*コンソールで実際に感じるもの*に変える。述べられた aim と、それに仕えると称するコードとの gap から、private なエンジンは **drift**(仕事とその述べられた purpose が離れた箇所)を surface し、記録を現実へ **reconcile** するのを助け、**あなたの注意に値する変化**を蒸留する ― 全部でなく、見るべき一つの交差点をあなたに届けるために。これらの機能が存在すること、そしてそれが Aim 構造で駆動されていることは public だ: それがコンソールの*目的*だから。それらが*どう*計算されるかは private な [`tmai-core`](https://github.com/trust-delta/tmai-core) エンジンに留まる ― charter が述べるのと同じ姿勢の下で(この秘匿が守るのは作者の motivation であって商用 IP ではない)。public な半分は*what*(目的地・到着・そして gap にあなたの注意を向けるツール); エンジンの*how* が private な半分だ。

## なぜこれは spec 駆動開発でないか

Aim と spec は形を共有する ― どちらも intent を書き留め、仕事をそれに照らして検査する ― だから違いを名指す価値がある。その違いは「done」の権威がどこに座るかだ。spec 駆動開発はそれを機械が検査できる受け入れ条件に座らせる: エージェントが条件 met とマークし、人が「それは実際には done でない」と言えば、修正は*より多くの条件*になる。その地平は、spec が十分完全になって agent-done = real-done になる点 ― つまり人間の満足が設計から排除された点だ。Aim は逆の前提から出発し、それを決して手放さない: **purpose が満たされたかは、人にしか判断できない。** あなたの「それは違う」は、patch で消すべき条件の gap ではない; それこそが全ての要点だ ― 設計が verdict を*置く*場所であって、消そうとする場所ではない。
